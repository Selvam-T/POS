# Main Status Footer

Updated: June 13, 2026

## Overview

The main window footer is the bottom operational status strip in `ui/main_window.ui`.
It is controlled by `modules/status_footer/status_footer.py` through
`MainStatusFooterController`.

The footer has three zones:

- `loggedInUserLabel` on the left: shows the current logged-in username.
- `statusbar` in the center: shows normal application StatusBar messages.
- `errorLogFrame` on the right: shows error-log state and actions.

`main.py` keeps a small compatibility override:

```python
def statusBar(self):
    custom_statusbar = getattr(self, 'statusbar', None)
    if custom_statusbar is not None:
        return custom_statusbar
    return super().statusBar()
```

This lets existing calls to `self.statusBar().showMessage(...)` keep working
while the visual message appears in the centered footer status area.

## Controller

`MainStatusFooterController` is bound from `MainLoader.__init__`:

```python
self.status_footer = MainStatusFooterController().bind(self)
```

After login, `main.py` updates the username with:

```python
window.status_footer.set_username(window.current_username)
```

## StatusBar Message Centering

Qt's native `QStatusBar` text is left-aligned by default. The controller adds a
centered `QLabel#statusMessageLabel` as a permanent widget inside the custom
status bar and wraps:

- `showMessage(message, timeout)`
- `clearMessage()`
- `currentMessage()`

This keeps existing application code compatible while making the visible message
appear centered between the left username label and the right error-log frame.

## Error Log Indicator

The footer watches `log/error.log` using `QFileSystemWatcher`.

State is based on whether the file exists and contains non-whitespace content:

- Empty log:
  - `errorLogStatusLabel` displays `No Error`.
  - `exportErrorLogBtn` and `clearErrorLogBtn` are disabled.
  - Buttons use `Qt.NoFocus`.
  - Dynamic property `hasError` is set to `false`.

- Non-empty log:
  - `errorLogStatusLabel` displays `Error Log!`.
  - `exportErrorLogBtn` and `clearErrorLogBtn` are enabled.
  - Buttons use normal focus.
  - Dynamic property `hasError` is set to `true`.

The dynamic property is applied to:

- `errorLogFrame`
- `errorLogStatusLabel`
- `exportErrorLogBtn`
- `clearErrorLogBtn`

`assets/main.qss` uses these selectors to switch between the quiet inactive
state and the visible active state.

## Export Behavior

The Export button writes a UTF-8 text copy of the current error log.

Export folder:

```text
~/POS_Exports/Error_Log
```

On Windows for the current user this is typically:

```text
C:\Users\SELVAM\POS_Exports\Error_Log
```

Filename convention:

```text
error_log_<dd><mon><yyyy>_<hh>-<mm>-<ss>.txt
```

Example:

```text
error_log_13jun2026_16-18-56.txt
```

The exported file includes:

- `Selvam POS Error Log`
- export timestamp
- source log path
- original log contents

After successful export, the centered StatusBar shows the export folder for
4000 ms, without the filename:

```text
Error log saved to C:\Users\SELVAM\POS_Exports\Error_Log
```

`errorLogStatusLabel` is not used for export success messages; it continues to
represent only the current log state (`No Error` or `Error Log!`).

## Clear Behavior

The Clear button truncates `log/error.log`.

After successful clear, the centered StatusBar shows:

```text
Error log cleared.
```

for 4000 ms. `errorLogStatusLabel` is refreshed back to `No Error`.

## Watcher Notes

`QFileSystemWatcher` should update the footer when `log/error.log` changes, but
it can be delayed or miss events when the file is edited through an IDE, replaced
by an editor save operation, or synced through OneDrive. The controller re-adds
the watched file and parent directory after file changes, export, and clear
operations.

Footer state checks use the log file size instead of reading the entire log.
This keeps the Clear action responsive even if the log becomes large. Clear also
refreshes the footer immediately after truncating the file. File-change
notifications are debounced briefly, and a lightweight one-second polling
fallback compares file size/mtime so missed watcher events still update the
right-side indicator.

## Smoke-Test Side Effects

Instantiating `MainLoader()` in tests or smoke checks runs normal startup code.
That includes database setup calls such as `ensure_cash_outflows_table()`.

If the database path is unavailable in the test environment, the existing app
startup path can write entries like:

```text
Cash outflows table ensure failed: unable to open database file
```

to `log/error.log`.

Those entries are generated by the normal startup error logger, not manually by
the footer controller. They can be cleared with the footer Clear button when
they are only test artifacts.
